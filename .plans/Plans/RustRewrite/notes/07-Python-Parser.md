# Phase 7 Debrief: Python Language Parser

## Decisions Made

- **`truncate_signature` consolidation bundled into Task 7.1**, not split into a separate pre-phase task. The Phase 5 debrief said Phase 6 was the consolidation moment; the Phase 6 debrief said "do it before Phase 7's parser body lands." The 7.1 brief turned that recommendation into a hard subtask: extract `truncate_signature` to `crates/codegraph-lang/src/helpers.rs` *first* (commit `071defe`), then scaffold the Python crate (commit `368c621`). Three byte-identical copies → one canonical implementation; cpp/rust/go re-export via `pub use codegraph_lang::helpers::truncate_signature;`. Test coverage consolidated to the highest-coverage set (8 tests). The forcing-function pattern worked — the consolidation had been deferred twice; making it a hard prerequisite for Python broke the cycle.
- **`find_enclosing_kind` consolidation deferred to Task 7.7**, not Task 7.1. By the time 7.1 shipped Python had a fifth byte-identical copy of the helper (cpp helpers + rust helpers + rust lib + go lib + python lib). The 7.7 implementer extracted it to `codegraph-lang::helpers::find_enclosing_kind` alongside `truncate_signature`. Five copies → one. The decision to defer until the *registration* task (rather than fold it into 7.1) was deliberate: bundling both helper extractions into 7.1 would have made that task too large; splitting them lets each consolidation land cleanly with its own test diff.
- **`PythonParser` accepts the default `resolve_call` and `resolve_include`** — does not override either. Same call as the Go and Rust plugins. The rationale is sharper for Python than for the static-typed siblings: Python's dynamic typing makes *any* static call resolution inherently noisy (a call to `obj.method()` could resolve to dozens of unrelated methods named `method` across the codebase), so the default scope-aware heuristic is the documented contract. Default `resolve_include` is a no-op for Python module paths because `from foo.bar import baz` records `foo.bar` as a dotted module path, not a filesystem path — the default basename match against the FileIndex correctly returns None for these.
- **Sequential dispatch for Wave 2 (7.2 → 7.3 → 7.4 → 7.5).** Four tasks all extending `parse_to_filegraph` with a new `extract_X` call; parallel work would have produced merge conflicts. Same convention Phase 5 and Phase 6 used. Now confirmed across three plugin phases — the orchestrator's task brief says "sequential" without further justification.
- **`from-form points at the module, not the imported name`.** The original Plans/PythonParser brief specified this rule; Phase 7's task 7.4 preserved it verbatim. `from typing import List, Dict` produces ONE edge with To='typing' (not two edges with To='List' and To='Dict'). An agent searching for "what does this file depend on?" wants modules, not symbols, and the imported names are already in scope after the import statement. The single-edge-per-`from`-statement contract is the load-bearing dependency-graph behavior.
- **`from __future__ import annotations` discovered to use a separate node kind mid-task.** During 7.4 implementation, tree-sitter-python 0.25 surfaced a separate node kind `future_import_statement` for `__future__` imports, distinct from the regular `import_from_statement` used for `from foo import bar`. The first iteration of the import query missed `__future__` imports entirely; a corpus test caught the regression. Fix: extend `IMPORT_QUERIES` to match both `import_from_statement` and `future_import_statement` with the same module-path extraction. Documented in the Phase 7 phase-doc verification field for 7.4 and as an anti-regression test.
- **Conditional imports (`if TYPE_CHECKING: import ...`) intentionally NOT extracted.** Tree-sitter sees these as block-level statements inside an `if_statement`, not as top-level `import_statement` nodes. The existing `IMPORT_QUERIES` deliberately don't enter `block` nodes inside conditionals. The decision: document the limitation in CLAUDE.md (Task 7.7), not work around it. Rationale: conditional imports are *intentionally* deferred imports — including them in the dependency graph would record edges that don't fire at module-import time, which contradicts the dependency-edge semantics of the rest of the graph. A regression test asserts these produce zero edges.
- **284 `requests` symbols vs. the brief's 400-1000 estimate.** Phase 7.6 ran `parse-test /tmp/requests/src/requests` (the published `src/requests` package only — tests, docs, and helper scripts intentionally out of scope per the brief). The actual count is 284. The brief's 400-1000 range was an estimate, not a contract; baseline-as-recorded is correct. The corpus regression test asserts ±10% against 284, not the brief's estimate. Recorded in `testdata/python/requests-baseline.txt` with a comment explaining the divergence so a future maintainer doesn't "fix" the count to match the prose.
- **`broken.py` syntax-error fixture extracts a usable symbol.** The corpus expected the parser to skip an error-recovery file gracefully (zero symbols, zero crashes). Tree-sitter-python's syntax-recovery behavior actually produces a valid `function_definition` for the partially-broken `def foo()` even when the body has ERROR children. Decision: accept this as the actual behavior; update the corpus test to assert exactly one symbol from `broken.py`, not zero. Validates the "tree-sitter is more recovery-aggressive than expected" assumption that later languages should inherit.
- **3-way cross-language collision test (C++/Go/Python) — Rust deliberately excluded.** The Phase 6 debrief recommended widening the 2-way `init` regression to 3-way. The 7.7 task did so. Rust was excluded on purpose: Rust's `init` would parse as an ordinary function and would add no new structural pressure; the load-bearing assertion is asymmetric isolation across distinct *languages*, not the pair count. Three positive assertions (each language's caller IS in its own init's callers) plus six negative assertions (no caller leaks into another language's init's caller list) — all hold simultaneously, all pin the `(Language, name)` SymbolIndex keying.
- **Quality scans landed at the END of Tasks 7.4, 7.5, 7.6, 7.7.** Each scan's findings folded into the next task's brief. By the end of the phase the cleanup is free; no separate cleanup commit needed beyond the comment fixes in this close-out commit. Two findings made it all the way to 7.8 (the 4-way/3-way comment text in `mixed_language.rs` and the misleading prose comment in `helpers.rs::find_enclosing_kind` test). Both are corrected as part of the close-out commit.

## Deviations

- **`requests` baseline came in at 284, not in the brief's 400-1000 range.** The brief's range was an estimate; the recorded value is correct. The regression gate is ±10% of 284, not of the prose number. Documented inside `testdata/python/requests-baseline.txt` so a future maintainer doesn't "correct" the file to match the brief.
- **Inheritance edge `from` field uses bare class names, not `path:Outer::Inner`.** The Phase 7 brief implied paths with parent qualifiers; the existing codebase contract (established in Phase 1 for C++ and Phase 5 for Rust) uses bare type names. The 7.5 implementer tested against the existing contract (correct) rather than the brief's prose (wrong) and surfaced the discrepancy in the 7.5 quality scan. The brief was edited to match shipped state. This is a recurring "spec lags codebase" pattern that the consolidated `truncate_signature` and `find_enclosing_kind` extractions should reduce.
- **`broken.py` extracts one symbol, not zero.** The brief expected zero (gracefully skipping ERROR-containing files); tree-sitter-python's recovery is more aggressive. The corpus test was updated to assert one symbol; this is the actual contract. Documented as part of the Python parser's behavior in the MANIFEST.
- **`__future__ import` separate node kind discovered mid-task 7.4.** Initial query iteration missed it; a corpus test caught the regression on the same task. The fix landed in the same commit as the rest of 7.4 import extraction; no separate fix-forward needed. Documented in the phase-doc verification field for 7.4.
- **Helper consolidations split across two tasks (7.1 for `truncate_signature`, 7.7 for `find_enclosing_kind`).** The Phase 6 debrief recommended a single pre-phase consolidation task. Splitting into "extract before scaffolding" + "extract before integration" kept each task's diff focused and let the test consolidation land alongside each helper. End result: one task per helper, both shipped, no follow-up needed.
- **The 7.7 quality scan flagged two minor comment inconsistencies** that did not block 7.7 from completing. Both are addressed in this 7.8 close-out commit (the `4-way`/`3-way` framing in `mixed_language.rs` and the misleading prose in `helpers.rs::find_enclosing_kind` test). The "carry findings forward" pattern means cleanup commits are essentially free — they ride on the next task's verification gates.

## Risks & Issues Encountered

- **Tree-sitter-python's `future_import_statement` separate node kind.** A real surprise mid-7.4: the initial import query matched `import_from_statement` (which covers `from foo import bar`) but NOT `future_import_statement` (which covers `from __future__ import annotations`). Caught by a corpus test; fixed by extending the query. The lesson generalizes: tree-sitter grammars regularly carve out special cases for syntactically-distinct constructs, and the "obvious" query may miss the special form. Mitigation: every import-form variant gets a corpus test, including the syntactically-special ones.
- **Test redundancy from spec-named test wave.** The 7.5 quality scanner flagged that the inheritance-test names were too closely matched to spec verbiage rather than to the assertion's intent. The 7.6 implementer renamed the redundant tests during corpus work. Cost: ~10 minutes; benefit: tests now read as "what they verify" rather than "what subtask they came from."
- **Tree-sitter syntax recovery surprise.** The `broken.py` fixture was supposed to verify graceful skipping; it actually demonstrates that tree-sitter extracts the still-recognizable parts of a partially-broken file. Two ways to read this: (a) "the parser is robust, ship it" (chosen path); (b) "the parser is too lenient and might extract bogus symbols from real-world syntax errors" (rejected — the cost of catching this in the wild is one bogus symbol per broken file, which is below noise). The decision was documented in MANIFEST.
- **Five copies of `find_enclosing_kind` at the time Phase 7 started.** The Phase 6 debrief flagged truncate_signature as the textbook over-due consolidation; in fact, by the time Phase 7 began there were *two* such helpers in 5+ identical copies. The consolidation in 7.1 (truncate_signature) and 7.7 (find_enclosing_kind) eliminated both. Lesson: phase debriefs that flag pending consolidations should *count the copies*, so the next phase brief sees the cost increasing each phase rather than a static "still pending."
- **Honor-system regression baselines.** `requests-baseline.txt` joins `logrus-baseline.txt` and the C++ baseline as `#[ignore]`-gated dogfood tests. Without a CI lane configured for network access (still not done), the baseline is effectively whoever last ran `--include-ignored`. Phase 6's debrief flagged this; Phase 7 inherits it. Worth a 30-minute CI configuration before SharedDaemon adds tenancy concerns.
- **Conditional-import suppression is a structural limitation, not a bug.** `if TYPE_CHECKING: import expensive_module` is a Python idiom for type-hint-only imports that don't fire at runtime. The decision to NOT include them in the dependency graph is correct — they're not dependencies in the operational sense — but it means a graph user looking for "all imports a file uses" will not see them. Documented in CLAUDE.md as a known limitation. A future enhancement could expose them as a separate edge kind (e.g. `TypeImports`); not in scope for Phase 7.

## Lessons Learned

- **The "consolidate when adding the Nth copy" pattern proved out twice.** `truncate_signature` shipped with three copies and was extracted in 7.1 just before the fourth would have been added. `find_enclosing_kind` shipped with five identical copies and was extracted in 7.7 right before integration tests added a sixth. Pattern: when you're about to add the Nth copy of a helper, the consolidation is mechanical and the test diff stays small. Defer past N=4 and the diff balloons because every plugin's test suite has drifted slightly.
- **The "carry quality findings forward" pattern is now a 5-fold-confirmed pattern.** Findings from 7.4 → 7.5 → 7.6 → 7.7 → 7.8, each one folded into the next task's brief with explicit "address these" items. By 7.8, only two trivial comment-text fixes remained (and they're folded into the close-out commit). Compare to a pattern of "scan once at the end" — those scans tend to surface 20+ findings that nobody addresses because the phase is "done." The carry-forward pattern keeps the deferral budget at zero.
- **Real-world dogfood with ±10% regression gate is now a 2-time pattern (logrus@v1.9.3 in Phase 6, requests@v2.32.3 in Phase 7) and works.** The brief's symbol count is always an estimate; the actual baseline is recorded in a `*-baseline.txt` file with a comment explaining any divergence. Future plugin work should follow the same shape.
- **Phase 5 → 6 → 7 brief refresh paid off again.** The 2026-04-30 plan-reviewer pass that pre-applied Phase 5/6 lessons to the Phase 7 brief meant zero cross-cutting fixes during implementation: object-safety test, strict version pin, default trait methods, watch-mode regression, Box+context registration, sequential-dispatch wave decision — all already correct in the as-shipped phase doc. The "refresh-brief-against-shipped-state" cadence has now paid back in three consecutive plugin phases.
- **Pre-phase consolidation as a hard subtask works better than as a recommendation.** The Phase 5 debrief said "Phase 6 is the consolidation moment." Phase 6 said "stay in scope, defer to Phase 7." Only when Phase 7's brief made the consolidation a hard subtask of 7.1 did it actually happen. Forcing functions beat reminders.
- **Asymmetric assertions are the load-bearing form for cross-language correctness tests.** The `cross_language_init_callers_stay_isolated` test asserts both "X IS in A's callers" AND "X IS NOT in B's callers" for every (A, B) pair. The negative half is what makes the test load-bearing — without it, a "callers includes everything in the codebase" bug would pass. Phase 6 established this pattern with 2-way; Phase 7 widened to 3-way; the same asymmetric form generalizes to N-way.
- **Tree-sitter grammars are more nuanced than the surface API suggests.** Each grammar has special node kinds for syntactically-distinct constructs that the "obvious" query misses (`future_import_statement` for Python; `function_signature_item` for Rust trait declarations; `method_elem` for Go interface method declarations). The mitigation is per-grammar: corpus tests for every variant, including the special forms. The lesson is general: never trust an extraction is complete until you've fed every dialect of the construct through a fixture.

## Skill Opportunities

### 1. `/planner:check-duplication` slash command — flags helpers with N≥3 byte-identical copies

- **What I observed twice in Phase 7**: `truncate_signature` was deferred for two phases; `find_enclosing_kind` accumulated to five copies before being extracted. Both deferrals were rational at decision time (scope discipline) but the cumulative cost grew silently. A scanner that runs at the end of each plan phase and reports "helper X has N copies in the workspace; consolidation cost is X lines + Y tests" would surface the cost continuously.
- **Where it belongs**: A new slash command that reads the workspace tree, finds functions with byte-identical or near-identical bodies in multiple crates, and emits a Markdown report grouped by helper.
- **Why a skill**: Duplicate-detection-via-eyeball is unreliable. The scanner is mechanical (AST-based or even SHA-based); the recommendation (consolidate now or defer to phase X) is what humans should make. Mechanizing the detection lets the deferral decision be informed.
- **Rough shape**: `/planner:check-duplication` scans `crates/*/src/**/*.rs`, hashes each `fn` body (post-fmt), groups by hash, reports groups with size >= 2. Output: a table of (helper name, crates, line count, test count). Run as part of `/planner:debrief` to catch the pattern earlier.

### 2. `/planner:dogfood-baseline` slash command — automates the real-world-validation gate

- **What I observed three times**: Phase 1 testdata_cpp baseline, Phase 6 logrus@v1.9.3 baseline, Phase 7 requests@v2.32.3 baseline. Each one followed the same pattern: clone external project at pinned tag, run parse-test, record the symbol count, gate the regression at ±10% with `#[ignore]` + `eprintln!+return` on missing fixture, document any divergence from the brief's prose estimate.
- **Where it belongs**: A new slash command that takes (language, project URL, pinned tag, expected range) and produces both the test-skeleton and the baseline file.
- **Why a skill**: The boilerplate (panic-on-missing-baseline-file but eprintln-on-missing-fixture; ±10% calc; `#[ignore]` discipline; baseline-file format) is identical and easy to get subtly wrong. Automating it eliminates the "spec said 400-1000, actual is 284" mismatch class of issue.
- **Rough shape**: `/planner:dogfood-baseline LANG=python PROJECT=psf/requests TAG=v2.32.3 PATH=src/requests` produces:
  - `crates/codegraph-tools/tests/dogfood_python.rs` skeleton with `#[ignore]`-gated test
  - `testdata/python/requests-baseline.txt` template (run once with `--include-ignored` to populate the actual count)
  - Documentation snippet for CLAUDE.md or the phase doc

### 3. `/planner:carry-forward` slash command — pre-populates the next task's brief from prior task's quality scan

- **What I observed five times**: 7.4 → 7.5 → 7.6 → 7.7 → 7.8, each implementer brief carried the prior task's quality-scanner findings as explicit "address these" items. The orchestrator did this manually. By the end of Phase 7 the cleanup was free; no separate cleanup commit needed.
- **Where it belongs**: A slash command that reads `<plan>/<phase>/notes/<prior-task-id>-quality-scan.md` (if it exists) and emits a "carry-forward findings" section for the next task brief.
- **Why a skill**: Manual carry-forward is reliable when the orchestrator remembers to do it. Automating it makes the pattern impossible to skip.
- **Rough shape**: `/planner:carry-forward <plan>/<phase>/<task-id>` reads the prior task's quality scan, extracts the Minor and Question findings, and pastes them as a "Address these" subtask list at the top of the next task's brief.

### 4. Phase debrief template field for "helpers consolidated this phase"

- **What I observed**: Phase 7's debrief lists two consolidations; future plan readers want to know which helpers shipped consolidated and which are still in N-copy state. Right now this lives in the Decisions Made narrative; it's not searchable.
- **Where it belongs**: An optional "Consolidations Landed" section in the debrief template (after Decisions Made, before Requirements Assessment).
- **Why a skill**: Debrief format consistency makes cross-plan queries possible ("which helpers got consolidated in which plan?"). Also creates a forcing function: a debrief that says "Consolidations Landed: none; pending: X (4 copies), Y (3 copies)" is louder than a narrative that buries the pending items.
- **Rough shape**: Template addition, no slash command needed. Update `shared/templates/debrief.md` to include the section.

### 5. Pre-phase plan-reviewer refresh as a documented cadence (carry-over from Phase 5/6 debriefs)

- **What I observed**: Phase 5 → 6 → 7 each got a plan-reviewer refresh against shipped state before the implementer dispatch. All three plugin phases shipped with zero cross-cutting fixes. The pattern is now confirmed three times.
- **Where it belongs**: `/planner:refresh-brief <plan>/<phase>` slash command, called as a hard prerequisite before `/planner:implement`.
- **Why a skill**: This carry-over has been pending since the Phase 5 debrief and is the highest-leverage process change still on the table. Three plugin phases shipped clean *because* of the refresh; not codifying the cadence means the next plan's first phase will skip it.
- **Rough shape**: Same as the Phase 6 debrief's proposal. The slash command reads the phase doc + the prior phase's debrief + the related design doc, runs `/planner:plan-reviewer` against the phase doc with the prior debrief as additional context, and outputs a Revise-or-Approve verdict.

### 6. Test-name-vs-test-intent linter

- **What I observed**: The 7.5 quality scan flagged that test names tracked spec verbiage rather than the assertion's intent. A pattern that would have caught this earlier is "after writing a test, ask whether the test name describes what fails when the test fails, not what subtask the test came from."
- **Where it belongs**: A `/simplify`-like pass that scans test names and surfaces ones that look like phase/task IDs or spec section numbers.
- **Why a skill**: The cost of fixing test names later (which Phase 7 did) is higher than the cost of catching them at write time. A linter pass catches the pattern early.
- **Rough shape**: A regex-based scanner over `#[test]`/`#[tokio::test]` function names looking for `phase_N`/`task_N_M`/`spec_section_X` patterns. Reports each as a candidate for renaming.
