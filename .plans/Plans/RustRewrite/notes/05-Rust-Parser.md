# Phase 5 Debrief: Rust Language Parser

## Decisions Made

- **`RustParser` accepts the default `resolve_call` and `resolve_include`** — does not override either. Same call as the C++ plugin. Default `resolve_include` (basename match against the FileIndex) is correctly a no-op for Rust `use` paths because they are dotted module paths, not filesystem paths; the wire format records the full path verbatim. Decision documented in code.
- **`macro_rules!` definitions explicitly produce zero symbols.** Decision baked into the queries module (the `DEFINITION_QUERIES` constant deliberately omits `macro_definition`) AND enforced by an anti-regression test. The phase doc's note about `macro_rules_definition` was a brief inaccuracy — actual tree-sitter-rust 0.24.2 node is `macro_definition`. Either way, the queries don't match it, and the test uses Rust source code (`macro_rules! foo { ... }`) so the assertion is grammar-name-agnostic.
- **`extern_crate_declaration` handled as `Includes` edges** — same kind as `use` edges. `extern crate alloc;` produces one edge with `to = "alloc"`. The `extern crate` keyword is rare in modern Rust but still appears in `no_std` crates.
- **`#[test] fn parser_is_object_safe_via_box_dyn()` is the canonical compile-time-interface check** for new language plugins, NOT the dead `fn _object_safety_check()` form. The latter would emit `dead_code` warnings that fail the `-D warnings` clippy gate; the `#[test]` form avoids the warning AND doubles as the `id() -> Language::X` assertion. Mirrors the shipped C++ test at `crates/codegraph-lang-cpp/src/lib.rs:542-545` exactly. This decision came from the plan-reviewer pass on the original 5.1 brief and is now established as the pattern for Phases 6 and 7.
- **Strict `=` version pin on `tree-sitter-rust = "=0.24.2"`** matching the Phase 1 C++ convention `tree-sitter-cpp = "=0.23.4"`. Guards against a patch release silently changing query node types or capture names. Implementer noted that `=0.24.0` does not exist on crates.io; used the latest `0.24.x` (`0.24.2`) per brief permission.
- **Sequential dispatch of wave 2 (5.2 → 5.3 → 5.4)** rather than parallel-with-worktrees. All three tasks modify `parse_file` (each adds a call to a new `extract_X` function), so parallel work would have created merge conflicts. Sequential dispatch is slower wall-time but eliminates the merge risk. Each task's commit cleanly builds on the prior; the diff history is readable.
- **`codegraph-parse-test` got `codegraph-lang-rust` as a dependency in Task 5.5**, not 5.6. The dogfood pass (a 5.5 deliverable) parses real `.rs` files, which requires the parse-test binary to register `RustParser`. The brief's "do NOT register in main.rs" guard targeted the MCP server's `main.rs` specifically; the parse-test binary is a separate developer harness whose entire purpose is to validate parsers against real codebases. Implementer made the right call.
- **`codegraph-tools` got `codegraph-lang-rust` as a dev-dependency in Task 5.5**, not 5.6. Required so `watch_rust_reindex.rs` could instantiate `RustParser` in its `fresh_server` helper. Mirrors the existing `codegraph-lang-cpp` dev-dep pattern.
- **`truncate_signature` was duplicated, not promoted to a shared crate.** The brief said "use the shared `truncate_signature` helper from `codegraph-core`" but no such shared helper existed — it lived only in `codegraph-lang-cpp/src/helpers.rs`. Implementer added a byte-identical copy to `codegraph-lang-rust/src/helpers.rs` rather than do a side-quest crate-restructure. Reasonable trade-off; the consolidation is now a known follow-up that should land naturally when Phase 6 (Go) needs the same helper.
- **`mod_item` is captured as an explicit no-op branch in `extract_definitions`**, not silently dropped. The verification field reads "mod_item populates Symbol.namespace recursively" — meaning OTHER symbols inside a mod block get `namespace = "a::b"`, NOT that `mod_item` itself produces a symbol. Implementer added `"mod.name" => {}` with a comment explaining the design choice rather than letting it fall through to the catch-all `_` arm. Surfaces the decision instead of hiding it.
- **`search_symbols` results do NOT carry a `language` field on the wire.** The Phase 1 wire format (`SymbolResult` in `crates/codegraph-tools/src/handlers/mod.rs`) deliberately omits the field for Go-byte compatibility. The mixed-language integration test in 5.6 therefore discriminates results by file extension via a small `language_from_file` helper. The handler-level `language` filter (`SearchSymbolsInput::language`) is still exercised end-to-end by all four search-by-language tests; the snapshot for `language=rust` confirms the filter narrows the result set to one entry. Brief mismatch; implementer's resolution preserved both the wire-format invariant and the test coverage.

## Deviations

- **`truncate_signature` duplicated rather than promoted to a shared crate (5.2).** The brief said `codegraph-core::truncate_signature`; that helper doesn't exist. Implementer added a byte-identical copy to `codegraph-lang-rust/src/helpers.rs` with three local unit tests covering the same edge cases the C++ test suite covers. Reasonable trade-off — promoting helpers to a new shared module mid-phase would have been scope creep. **Future fix:** when Phase 6 (Go) lands, both Rust and Go will have copies of this helper; that's the natural moment to extract to `codegraph-lang::helpers` (or similar shared module).
- **Brief node-type name inaccuracies surfaced and self-corrected:**
  - 5.2: brief said `macro_rules_definition`; actual tree-sitter-rust 0.24.2 node is `macro_definition`. Crate-level docstring corrected to use the actual name. The `DEFINITION_QUERIES` constant doesn't match either name (intentional exclusion), and the anti-regression test uses Rust source code rather than the AST node name, so the test still proves the property regardless.
  - The original 5.1 brief said "compile-time interface check `fn _object_safety_check() { ... }`" referencing C++ `lib.rs:543`. Line 543 in the shipped C++ file is actually the `#[test]` form, not a const-fn or dead-fn. The plan-reviewer caught this before implementation; the revised brief now mandates the `#[test]` form explicitly.
- **`codegraph-lang-rust` registered in `codegraph-parse-test` during 5.5, not 5.6.** The 5.6 brief carefully said "do NOT register in main.rs" — meaning the MCP server's `main.rs`. The parse-test binary is a separate harness. Implementer added it during 5.5 because the dogfood-validation gate was meaningless without it. Reported as a deviation; reasonable.
- **`codegraph-lang-rust` added as dev-dep on `codegraph-tools` during 5.5.** Required for `watch_rust_reindex.rs` to instantiate `RustParser`. Mirrors the existing `codegraph-lang-cpp` pattern. Reasonable.
- **Sequential dispatch for wave 2 (5.2/5.3/5.4) rather than parallel-with-worktrees.** Original wave-dispatch convention from `/planner:implement` is parallel. Orchestrator chose sequential because all three tasks modify `parse_file` (each adds a call to a new `extract_X` function); parallel work would have produced merge conflicts. Total wall time: ~32 minutes for the three tasks; estimated parallel wall time: ~15 minutes minus merge effort. Trade-off worth it — every commit is clean and the diff chain reads naturally.
- **No commit landed for Task 5.7.** All gates green on the as-shipped state; no fix needed; no commit produced. The phase doc verification field is satisfied by the `cargo` invocations themselves, not by an explicit commit.
- **Brief said `tree-sitter-rust = "=0.24.0"`; actual pin is `=0.24.2`.** `0.24.0` doesn't exist on crates.io. Brief explicitly authorized "use the latest 0.24.x and pin it strictly" as a fallback. Pin preserved.

## Risks & Issues Encountered

- **Brief node-type names drifted from the pinned tree-sitter-rust version.** Two surface-level mismatches (`macro_rules_definition` vs `macro_definition`; `lib.rs:543` referenced the wrong style of compile-time check). Both self-corrected by implementers cross-checking against the actual grammar's `node-types.json` and the actual C++ source. Caught by quality-scanner and plan-reviewer respectively; no actual bugs shipped. Worth flagging: phase briefs that name AST node types or cite line numbers should be re-verified at implementation time.
- **`parse_file` is a single-file conflict point.** Sequential dispatch (5.2 → 5.3 → 5.4) avoided it; parallel dispatch with three agents touching the same function would have required worktree isolation + manual merge. Future plugin phases (6, 7) have the same shape and should follow the sequential pattern.
- **`truncate_signature` divergence between C++ and Rust copies is now a quiet drift risk.** Two copies exist; if a UTF-8 boundary fix lands in one, the other won't see it. Mitigation: the unit tests in both crates are byte-identical. Real fix: extract to a shared module (deferred to Phase 6).
- **`search_symbols` wire-format mismatch with the brief's mental model.** The brief implied results carry a `language` field; they don't (Phase 1 deliberately omitted it for Go-byte compatibility). The integration tests in 5.6 work around this with a file-extension-to-language helper. Not a real bug — just a brief that didn't reflect the wire-format invariant. Fixed at implementation time.
- **`#[allow(dead_code)]` carried as scaffolding from 5.1 → 5.4.** Each phase's brief explicitly required removal of the prior phase's `#[allow]` annotations once the field/helper got wired. Implementers tracked this carefully; final state has zero `#[allow]` in `crates/codegraph-lang-rust/`. Worked, but the pattern is fragile — a forgotten `#[allow]` would have shipped a `-D warnings` failure that would only surface at the next clippy run.
- **Dogfood pass on the workspace itself was a load-bearing validation.** Parsed 43 files / 835 symbols / 7651 edges with zero warnings; spot-checked the `LanguagePlugin` trait extraction and the `CppParser`/`RustParser` inherits edges. Without this, a real-world parsing bug could have shipped without anyone noticing until the user tried to use the plugin against their own Rust codebase.

## Lessons Learned

- **Pre-implementation plan refresh against shipped state pays off.** Two rounds of plan-reviewer on the Phase 5/6/7 docs caught: missing `Box::new(...)` registration, missing `Cargo.toml` dependency subtask, missing `id() → Language::X` requirement, missing watch-mode reindex regression, dead-`fn` compile-time-check that would have failed `-D warnings`, missing `extern crate` handling, missing `where`-clause generics fixture, missing `testdata/mixed/` cross-language search fixture. Every one of these would have surfaced as either a build failure or an incomplete acceptance test if shipped to implementation as-drafted. Total reviewer time: ~5 minutes; saved at least one fixup commit per finding.
- **The `#[test] fn parser_is_object_safe_via_box_dyn()` form is the canonical compile-time interface check** for new language plugins in this codebase. Mirrors the shipped C++ test at `lib.rs:542-545`. Doubles as the `id() → Language::X` assertion. The dead-`fn _object_safety_check()` alternative emits a `dead_code` warning that fails `-D warnings`. The `const _: fn() = || { ... }` IIFE alternative compiles but is structurally less natural than a real test. Pin this pattern in the next phase briefs.
- **Sequential dispatch is correct when tasks share a single file.** The wave-2 tasks (5.2/5.3/5.4) all extend `parse_file` with a new `extract_X` call. Parallel dispatch with worktree isolation would have required a merge step; sequential dispatch eliminates that. Time cost is small (~17 minutes extra wall time), risk reduction is real. Phase 6 and 7 should follow this pattern for the equivalent waves.
- **Dogfood validation against the project's own source is high-leverage.** The 5.5 dogfood pass parsed 43 files and 835 symbols without any synthetic test fixture — a real workload that exercises edge cases the corpus tests can't anticipate. Phases 6 and 7 don't have a workspace-internal dogfood target (no Go or Python in this repo), but they have pinned external projects (logrus@v1.9.3 for Go; requests@v2.32.3 for Python) that serve the same purpose.
- **Brief node-type names drift from grammar versions.** Tree-sitter grammar versions evolve; node names can change between major releases. Briefs that name AST nodes should be re-verified against the pinned version's `node-types.json` at implementation time. Implementers in this phase did this implicitly; making it explicit in future briefs would prevent the small surprise.
- **Anti-regression tests pay back immediately.** The `trait_impl_method_parent_is_type_not_trait` test (5.2) and `call_inside_trait_impl_method_has_type_qualified_from_not_trait` test (5.4) both exist because the trait-impl disambiguation is the easiest place to write a wrong implementation. The `macro_rules_definition_produces_zero_symbols` test exists because the deliberate exclusion is the kind of thing a future implementer might "fix" by adding the node type to the queries. These tests are cheap insurance against the most plausible future regressions.
- **Wire-format snapshots are still the contract of record.** Phase 5.6 added 4 new `.snap` fixtures; the existing 35 from Phase 3 remained unchanged. Snapshot tests caught zero issues in this phase but proved that Rust integration didn't drift any of the existing tool surfaces. The discipline is paying off.
- **The plan-reviewer feedback loop is not free, but it's worth it.** Two rounds of review against the originally-drafted Phase 5 doc surfaced the cross-cutting fixes that then applied cleanly to Phases 6 and 7 too. The third reviewer pass after revision returned Approve for Phase 5 and clean Revise for 6/7. Total cycle: ~10 minutes; alternative was rolling discoveries during implementation that would have created more fixup commits.

## Impact on Subsequent Phases

- **Phase 6 (Go) and Phase 7 (Python) briefs already incorporate Phase 5 lessons.** The pre-implementation refresh on 2026-04-30 pushed all the cross-cutting fixes (Box::new registration, Cargo.toml gate, id() requirement, `#[test]` object-safety, strict `=` pin, watch-mode reindex regression, default `resolve_call`/`resolve_include` decision, `extensions()` lowercase) into the 6 and 7 phase docs. They should ship without revisiting these issues.
- **`truncate_signature` consolidation should land in Phase 6.** When Go needs the same helper, the natural moment to extract it to a shared module (`codegraph-lang::helpers` is the most likely home) arrives. Three copies is the threshold to consolidate; we'll be at three (C++, Rust, Go) at that point.
- **Sequential dispatch for parse_file-shared tasks.** Phase 6 wave 2 (6.2 definitions, 6.3 calls, 6.4 imports) has the same shape as Phase 5 wave 2 — three tasks all modifying `parse_file`. Use sequential dispatch. Same for Phase 7 wave 2 (7.2 definitions, 7.3 calls, 7.4 imports, 7.5 inheritance — four tasks in this case).
- **`codegraph-tools` already has `codegraph-lang-rust` as a dev-dep.** Phase 6 needs to add `codegraph-lang-go` as another dev-dep for `watch_go_reindex.rs`. Phase 7 adds `codegraph-lang-python` for `watch_python_reindex.rs`. Pattern is established.
- **`codegraph-parse-test` already registers RustParser.** Phase 6 needs to extend it to register GoParser; Phase 7 extends to register PythonParser. The "register in parse-test during the corpus task, register in MCP server during the integration task" split is now established.
- **`testdata/mixed/` has `foo.cpp` + `foo.rs` + a shared `helper` anchor.** Phase 6.6 extends it with `foo.go`; Phase 7.7 extends with `foo.py`. The cross-language-search test grows from 2-way (5.6) to 3-way (6.6) to 4-way (7.7) along the same lines.
- **`#[test] fn parser_is_object_safe_via_box_dyn()` is the canonical pattern** for new plugins. Phases 6 and 7 should use it without further reviewer prompting.
- **Watch-mode reindex regression test pattern is set.** `watch_rust_reindex.rs` (5.5) is the template; Phase 6's `watch_go_reindex.rs` and Phase 7's `watch_python_reindex.rs` should follow the same shape — modify a fixture, verify symbol-set update, verify no dangling edges. Phase 7 additionally exercises both Inherits and Calls edge pruning per the revised brief.
- **`SharedDaemon` plan (deferred draft in `Designs/SharedDaemon/`) inherits a 2-language working binary now.** It will eventually inherit a 4-language one once Phases 6/7 ship.

## Skill Opportunities

- **`make new-plugin LANG=...` Makefile target** to scaffold a new language plugin crate. Generates `crates/codegraph-lang-<LANG>/Cargo.toml` (with workspace-deps stub), `src/lib.rs` (with the `#[test]` object-safety + id() check + cached Query objects + `LanguagePlugin` impl skeleton), `src/queries.rs` (empty constants), `src/helpers.rs` (the per-language helpers stubbed). Phases 6 and 7 will both want this; the existing Phase 5 work is the template. Belongs in the project Makefile or as a shared script under `scripts/`. Inputs: `LANG=rust|go|python|<future>`. Outputs: a compiling-but-empty plugin crate that passes `cargo build -p codegraph-lang-<LANG>` immediately. When to invoke: at the start of every new language plugin phase.
- **Promote `truncate_signature` to a shared module.** Currently duplicated in `codegraph-lang-cpp/src/helpers.rs` and `codegraph-lang-rust/src/helpers.rs`. Phase 6 will need a third copy. Move it to `codegraph-lang::helpers` (or similar) before the third copy lands. Belongs as a small refactor PR — could be a dedicated `/planner:simplify` invocation if such exists, otherwise a minor pre-Phase-6 task.
- **`make audit` Makefile target wrapping `cargo audit` install-if-missing.** Phase 4 debrief flagged this; Phase 5 ran `cargo audit` directly and worked. A one-liner Makefile target would replace the manual install dance with `command -v cargo-audit || cargo install --locked cargo-audit; cargo audit`. Easy win for Phase 6/7's structural verification gates.
- **Make `make check` aggregate fmt + clippy + test + audit.** Phase 4 debrief raised this. Phase 5 ran `cargo fmt --check && cargo clippy --workspace --all-targets -- -D warnings && cargo test --workspace && cargo audit` four times — once per task at minimum. A `make check` Makefile target would replace four commands with one and document the canonical pre-commit gate. Belongs in the project Makefile. Useful in any phase, every commit.
- **Document the sequential-dispatch heuristic for `/planner:implement`.** Currently the orchestration choice (parallel vs sequential per wave) lives in the orchestrator's head. The heuristic is: "if multiple tasks in a wave modify the same file, dispatch sequentially; if they modify different files, dispatch in parallel (with worktree isolation if needed)." Worth documenting in `/planner:implement` skill instructions or as a project convention. Saves the next orchestrator (human or LLM) from having to rediscover it.
- **A grammar-version cross-check at the start of each plugin phase.** Run `cat node-types.json | jq '.[] | .type' | grep -i <expected-name>` to confirm the brief's named node types exist in the pinned grammar version. Phase 5 had two small mismatches (`macro_rules_definition` → `macro_definition`; line-number cross-references) that surfaced during implementation. A pre-implementation grammar-check would catch these in the brief stage. Could be a tiny script under `scripts/check-grammar-nodes.sh` invoked by the brief author.
- **The `/planner:plan-reviewer` → apply findings → re-review pattern is reusable.** Phase 5 went through two full reviewer cycles before any code landed. The pattern: dispatch reviewer with the spec doc + as-shipped state context → apply findings → dispatch reviewer again with revised doc → if Approve, dispatch `/planner:implement`. Worth formalizing in the `/planner:plan` skill flow as the recommended cadence for any phase doc that hasn't been reviewed since the prior phase shipped.
