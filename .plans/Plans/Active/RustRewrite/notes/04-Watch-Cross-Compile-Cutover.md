# Phase 4 Debrief: Watch Mode, Cross-Compile & Go Cutover

## Decisions Made

- **`WatchHandle` holds `Debouncer<RecommendedWatcher, RecommendedCache>`, not the design-spec'd `FileIdMap`.** notify-debouncer-full v0.7 renamed/rebranded its canonical cache type to `RecommendedCache`. The design doc was written against an older signature; the implementer used the v0.7-canonical type and noted it in the phase doc. Same behavior, current API.
- **`tokio::sync::Mutex` for `index_lock`, `parking_lot::RwLock` for graph state.** The index lock is held across `await` boundaries (analyze does async file I/O); a parking_lot mutex would poison if it crossed an await. The graph's RwLock is short-lived, never crossing await â€” parking_lot is the right pick there.
- **Watch-event "drop on contention" rather than queue.** When `index_lock.try_lock` fails (an analyze is in flight), the event is dropped, not queued. Queuing would let a busy editor session OOM the watcher; the in-flight analyze will pick up the file's current state on its next pass anyway. This was already in the design â€” Phase 4.2 just enforced it.
- **Single write-lock guard on `inner.watch` in `watch_start`.** First Phase 4.1 commit (`8e8c0cc`) used a read-then-write pattern that allowed two concurrent `watch_start` calls to both pass the "is None?" check, then both install handles. Quality-scanner caught it as a TOCTOU race. Fixed in `7c2a065` with a single `let mut watch_guard = inner.watch.write()` covering both the check and the install. Barrier-driven regression test makes the race deterministic instead of probabilistic.
- **`Graph::prune_dangling_edges(removed_ids)` to fix cross-file edge invariant.** After `Graph::remove_file_unsafe(A)` cleared `nodes["A:old_fn"]`, `adj["B:caller"]` still held `target: "A:old_fn"` because the edge's `file == B`. Quality-scanner caught it on commit `7e947ad`; fixed in `3d2bd38` by computing `pre_existing - new_set` and pruning matching adjacency entries. New regression test (`watch_dangling_edges.rs`) fails without the fix.
- **`spawn_blocking` wrapping the read+parse in `try_reindex_file`.** First Phase 4.2 commit ran `std::fs::read` plus `parse_file` directly on the tokio worker. That blocks the runtime under load. `3d2bd38` moved both into `tokio::task::spawn_blocking`. Same Minor finding flagged the cached-config invariant (`let _ = inner.config.read().clone()`) which the comment now documents.
- **Watch reindex uses the cached `inner.config`, not a fresh `<root>/.code-graph.toml` re-read.** Re-reading the file on every event would surprise users who edit `.code-graph.toml` mid-session â€” and would slow the hot path with disk I/O. The cached-config invariant: whatever `analyze_codebase` loaded is what watch events use, until the next `analyze_codebase` reloads it. Unit test asserts a config injected onto `ServerInner` after analyze is observed by the watch loop.
- **Go cutover: completed plans stay unchanged, draft/planned plans get marked superseded.** Mid-Phase-4.4 user directive: "old plans that reference Go which are COMPLETED do not change." Draft Go-targeted plans (`Plans/{Go,Python,Rust}Parser/`, status `draft`/`planned`) â†’ superseded with forward-link. Completed Go plans (`Plans/CodeGraphMCP/`, status `complete`) and implemented designs (`Designs/LLMOptimization/`, status `implemented`) â†’ left alone as historical record.
- **README rewritten in Phase 4.5 as the canonical project README.** Replaced the Go-era version. Sections: install (`cargo install` or `make release` per platform), MCP client config snippet, `.code-graph.toml` schema, all 15 tools listed (corrected `generate_diagram` from brief's `generate_mermaid`), C++ limitations, build commands. `.code-graph.toml.example` shipped at repo root.
- **Cross-compile via cargo-zigbuild reverted at end-of-phase (commit `9c85d68`, 2026-04-30).** The deferred multi-platform pass (per user "defer the multi-platform build until last") built four targets cleanly (linux x86_64-gnu/musl, aarch64-musl, windows-gnu, all under 9 MB) but failed both macOS targets â€” cargo-zigbuild needs an Apple-framework-bearing macOS SDK at link time (`CoreFoundation`, `CoreServices`) and zig itself doesn't synthesize one. Two paths: vendor a macOS SDK on every build host (operational complexity + Apple licensing question), or drop cross-compile entirely. User chose the latter: `make release` is now a host-build alias; native per-platform builds are the supported path. `[profile.release]` (strip + thin LTO + codegen-units=1) was retained because it benefits the host build too.

## Deviations

- **Cross-compile pipeline shipped, then removed.** The original 4.3 brief specified all 6 platform binaries from one Linux host via cargo-zigbuild. The Phase 4.3 commit (`1fc2f3a`) shipped *infrastructure only* â€” Makefile recipes, `[profile.release]`, `.github/workflows/release.yml`, README section â€” per a user mid-phase decision to defer the actual multi-platform invocations until the end of the phase. When that deferred pass ran on 2026-04-30, the macOS targets failed at link time on Apple framework symbols. User decision: revert the entire cross-compile path rather than vendor a macOS SDK. Result: `Makefile`, `README.md`, `CLAUDE.md`, and `.github/workflows/release.yml` shrank back to the host-only path; `[profile.release]` retained.
- **`make release-tar` recipe added in 4.5, then removed in 4.3-revert.** Same fate as the rest of the cross-compile infra. Per-platform builders can `tar -czf â€¦` directly.
- **`WatchHandle` holds `RecommendedCache`, not the design's `FileIdMap`.** API rename in notify-debouncer-full v0.7. No behavior difference; the design doc reads as if `FileIdMap` is the canonical type, which is no longer true on the pinned version.
- **Two quality-scanner Major findings landed as fixup commits, not as blocking gates on the original task commit.** Phase 4.1 race fix (`7c2a065` after `8e8c0cc`); Phase 4.2 dangling-edges + blocking-I/O fix (`3d2bd38` after `7e947ad`). Both Majors caught real bugs the implementer missed; the fixup-commit pattern kept history readable (each commit documents one concern) at the cost of a slightly noisier log.
- **4.4 disposition rule narrower than brief.** Brief originally said "supersede all old plans/designs." User clarified: only draft/planned artifacts move to superseded; completed/implemented ones stay as historical record. Implementer carved out `Plans/CodeGraphMCP/` (status `complete`) and `Designs/LLMOptimization/` (status `implemented`).
- **Sample config schema mismatch in 4.5 brief.** Brief drafted `extra_ignores`/`exclude_globs`; actual `RootConfig` uses `extra_ignore`/`respect_gitignore`/`follow_symlinks`. Implementer deferred to code (per brief's instruction). `.code-graph.toml.example` matches the actual struct.
- **Tool name typo in 4.5 brief.** Brief said `generate_mermaid`; actual tool is `generate_diagram`. README uses the correct name.

## Risks & Issues Encountered

- **TOCTOU race on `inner.watch` in `watch_start`.** Read-then-write pattern allowed two concurrent `watch_start` calls past the "is None?" check. Both would install handles; one would silently leak its debouncer. Caught by quality-scanner on commit `8e8c0cc`. Fixed by single write-lock guard. Barrier-driven regression test makes the race deterministic.
- **Dangling cross-file edges after rename.** `Graph::remove_file_unsafe(A)` cleared the *nodes* but left *edges* in `adj[B]` pointing into removed `A` symbols. Any subsequent `get_callees(B)` would return stale targets. Caught by quality-scanner on commit `7e947ad`. Fixed by `Graph::prune_dangling_edges(removed_ids)`. Regression test in `watch_dangling_edges.rs` would fail without the fix.
- **Blocking I/O on tokio worker thread.** `std::fs::read` + `parse_file` ran inside `try_reindex_file` without `spawn_blocking`. Caught by quality-scanner; Minor severity. Fix wrapped both in `spawn_blocking` so a slow-disk reindex doesn't stall the runtime.
- **macOS cross-compile blocker (cargo-zigbuild + Apple SDK).** The deferred end-of-phase build pass surfaced the SDK requirement only after 4.3 had already shipped infrastructure and a CI workflow. Either path forward had cost: vendoring an SDK is operationally heavy and licensing-risky; reverting throws away the infrastructure work. User chose revert. Net: Phase 4.3 ships ~272 lines of code that were later removed; the time spent designing the cross-compile recipes and CI workflow is sunk.
- **Toolchain blocker for the deferred 4.3 pass.** Build host lacked rustup, zig, and cargo-zigbuild at the time 4.3 was originally attempted. User explicitly chose to defer rather than authorize sudo install. When the toolchain became available later, the macOS blocker surfaced. Both choices were correct in isolation; their combined effect was the wasted infrastructure work above.
- **Plan brief said `FileIdMap`; pinned dep had `RecommendedCache`.** Implementer noticed the discrepancy and used the pinned dep's canonical type. Low-risk because the types are interchangeable for our use case, but the design doc should be updated if anyone re-reads it.
- **`.github/` directory orphaned after workflow removal.** Manual `rmdir` to clean up the empty parent dirs after `rm` of the workflow file. Not a real issue, but `git rm` followed by `git add .` would have been one step instead of two.

## Lessons Learned

- **"Defer multi-platform builds until last" was the right call by the user, even though it surfaced the blocker after 4.3 had shipped.** The alternative â€” gating 4.3 on a successful 6-platform build â€” would have stalled the entire phase on a toolchain install dispute. Deferring let 4.4 (Go cutover) and 4.5 (release readiness) ship on the assumption that 4.3 was infrastructure-ready, then cleanly revealed the blocker without holding up the phase. The infrastructure work being thrown away is the cost of optionality. Worth it.
- **Cross-compile via zig hits a wall at Apple frameworks.** zig synthesizes a libc and a linker; it cannot synthesize Apple's SDK headers and frameworks. Any Rust crate that links to `CoreFoundation`/`CoreServices` (which `notify` and several common deps do) needs a real macOS SDK. For multi-target distribution that includes Mac, the choices are: (a) vendor an SDK on the build host, (b) use a macOS runner in CI, (c) drop cross-compile and build natively. (c) was chosen here because the project doesn't publish prebuilt binaries â€” users `cargo install` from source. For projects that *do* ship binaries, (b) via GitHub Actions matrix is the standard answer.
- **Quality-scanner Major findings on watch concurrency caught two real bugs the implementer missed.** Both the TOCTOU race and the dangling-edges invariant were subtle â€” passing a quick read-through, surfacing only under concurrent or sequenced operation. Per-task quality-scanner is high-leverage on concurrency-heavy code.
- **The `cached-config` invariant in `try_reindex_file` is a load-bearing convention worth documenting.** Re-reading `<root>/.code-graph.toml` on every watch event would (a) surprise users who edit the file mid-session and (b) add disk I/O to the hot path. The cached invariant says: whatever `analyze_codebase` last loaded is what watch uses. The `let _ = inner.config.read().clone()` line plus the explanatory comment in code is how it's enforced + documented.
- **Plan briefs lock the version of dependencies; check them.** The `FileIdMap` â†’ `RecommendedCache` rename is a small example of how a brief written against a documented API can drift from the pinned version. Not catastrophic here, but worth a quick `cargo doc --no-deps` or `cargo doc --package <dep>` check during implementation.
- **The user's "completed plans don't change" rule prevented rewriting historical record.** A literal reading of the 4.4 brief would have marked `Plans/CodeGraphMCP/` (status `complete`) as superseded too, retroactively rebranding shipped Go work as obsolete. The user's clarification preserved the rule that completion is final â€” superseded is for *intent that was never realized*, not for *work that was completed and is now succeeded by a newer effort*.
- **Per-task fixup commits (e.g. `7c2a065`, `3d2bd38`) keep history readable.** The alternative would have been a single massive Phase 4.2 commit with the original code + all the fixes squashed. The split makes the bug+fix pairs easy to find via `git log` and makes `git revert` of just the fix cleanly possible if it ever became needed.
- **Phase 4 was the cutover boundary, and the cutover commit shipped clean.** No co-existence period, no feature flag, no parallel-run-and-compare. After commit `6315847`, Go is gone. This was always the design's intent and it worked: zero rollback path was needed.
- **Watch + analyze concurrent test (`watch_race.rs`) is the gold standard for concurrency tests in this codebase.** Spawn two real loops on a real `TempDir`, run them in parallel, assert no panics + every query returns a coherent snapshot. Not a unit test, not a snapshot test â€” an integration test that exercises the actual race surface. Worth replicating wherever a new lock is introduced.

## Impact on Subsequent Phases

- **Phases 5/6/7 (Rust/Go/Python plugins) inherit a stable watch mode.** New plugins just need to register their extensions and implement `LanguagePlugin`; the watch loop dispatches by `registry.for_path` automatically. No watch-related code changes needed in those phases.
- **No CI release pipeline.** Phase 5/6/7 plugin authors don't get an automated multi-platform build on PR or tag. If we ever publish prebuilt binaries (currently a TODO in the README), the design will be revisited â€” likely a GitHub Actions matrix with one job per native runner (linux, macos, windows) rather than zig-based cross-compile.
- **`make release` is the canonical build alias.** Documented in CLAUDE.md, README, and Makefile. Phase 5/6/7 should use it (or `cargo build --release`) and not assume any cross-compile recipe exists.
- **Cached-config invariant for watch.** Phase 5/6/7 plugins should not assume `.code-graph.toml` is re-read on every event. If a plugin needs config-derived state, it should snapshot at `parse_file` time using the config the indexer passed in, not re-read the TOML.
- **`Graph::prune_dangling_edges` is part of the public API.** Phase 5/6/7 plugins that emit cross-file edges benefit from this fix â€” the plumbing in `try_reindex_file` already calls it. Plugin authors don't need to do anything; just be aware that the semantics for "edges into removed symbols" are now "pruned automatically on remove or modify."
- **`SharedDaemon` plan (deferred draft in `Designs/SharedDaemon/`) is unblocked.** Its starting point is "a working, validated multi-language Rust binary." Phase 4 delivers the validated, multi-binary-but-not-yet-multi-language Rust binary; Phases 5/6/7 add the languages, then SharedDaemon picks up.

## Skill Opportunities

- **`make check` aggregating fmt + clippy + test.** Repeatedly ran `cargo fmt --all --check && cargo clippy --workspace --all-targets -- -D warnings && cargo test --workspace` as separate commands during Phase 4 (and prior phases). A single `make check` Makefile target would replace three commands with one and would document the canonical pre-commit gate. Belongs in the project Makefile. Inputs: none. Output: pass/fail. When to invoke: before every commit; in CI; in `/implement` per-task verification.
- **`make audit` wrapping `cargo audit` install-if-missing.** Phase 4.5 manually installed `cargo-audit` then ran the scan. A Makefile target that does `command -v cargo-audit || cargo install --locked cargo-audit` then `cargo audit` would make this a one-step gate. Belongs in the project Makefile. Useful in any phase that touches `Cargo.toml`.
- **Per-task quality-scanner agent dispatch on concurrency-heavy code.** Phase 4.1 and 4.2 both surfaced Major findings via quality-scanner. The pattern is: any task that touches a lock, an `await` boundary, or an event-loop should auto-dispatch quality-scanner before the commit lands. The `/implement` skill already supports this for per-task review; documenting "concurrency tasks always get quality-scanner" as a project convention would reduce the chance of a TOCTOU/dangling-state bug shipping.
- **Race regression test pattern (`std::sync::Barrier` for deterministic concurrent entry).** Used in Phase 4.1's `watch_start` race fix. The pattern: spawn N threads, each blocks on a barrier, then proceed concurrently â€” making the race deterministic instead of probabilistic. Worth elevating as a project-wide convention for any "two callers race on shared state" test. Could be a tiny helper in a future `crates/codegraph-test-utils` crate.
- **Deferred end-of-phase verification pass.** The "ship infrastructure now, run the destructive validation last" pattern from 4.3 is generally useful when a verification step requires authorization (sudo install, paid API call, network access, etc.) that the implementer can't grant unilaterally. Could be a `/planner:defer-verification` slash command that the implementer dispatches when they hit a permission wall, rather than blocking the task. Output: an item in the phase's "Acceptance Criteria" with `[deferred]` plus a documented prerequisite list.
- **Plan-doc / README sync check.** The plan README's `phases[].status` and the phase doc's frontmatter `status` drifted out of sync (4.3 fixup) more than once during Phase 4. Could be a `/planner:lint` or `make plan-lint` that walks `Plans/Active/*/README.md` and diffs against each phase doc. Belongs in the planner plugin (currently has no equivalent). Useful at the end of every phase before debrief.
- **Sample-config / actual-struct sync check.** `.code-graph.toml.example` was almost shipped with the wrong field names because the brief drafted them from memory rather than from `RootConfig`. Could be a derive macro on `RootConfig` that emits the example file at build time, or a `cargo test` that round-trips `.code-graph.toml.example` through `toml::from_str::<RootConfig>` and asserts no unknown keys. The latter is probably the right shape â€” minimal new code, fails loudly on drift.
