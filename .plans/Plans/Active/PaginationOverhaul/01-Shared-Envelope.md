# Phase 1: Shared Page<T> envelope + search_symbols migration

## Overview

Establish the shared pagination envelope before touching any other tool. Promoting `search_symbols`'s private `SearchResponse` to a generic `pub struct Page<T>` is the cheapest first move and the strongest guard against drift: if the migration changes the wire format, existing `search_symbols` snapshots will fail loudly, and the rest of the plan stops until it's fixed.

Zero behavioral change is the success criterion. Every existing test and snapshot passes; the only diff is type names.

## 1.1: Add `Page<T>` generic envelope in handlers/mod.rs

### Subtasks
- [x] Add `pub struct Page<T> { pub results: Vec<T>, pub total: u32, pub offset: u32, pub limit: u32 }` to `crates/codegraph-tools/src/handlers/mod.rs`
- [x] Derive `Debug` and `Serialize`; constrain `T: Serialize`
- [x] Match the field-declaration order of the existing `SearchResponse` exactly — `serde` emits fields in declaration order; the existing search_symbols snapshot is the ground truth for the resulting JSON shape

### Notes
The existing `SearchResponse` in `symbols.rs:59-65` is the model. Field types must match exactly — `u32`, not `usize`, to preserve the wire format byte-for-byte. The struct lives in `handlers/mod.rs` because every paginated handler is in a sibling module (`symbols.rs`, `structure.rs`, `query.rs`).

Doc-comment correction (post-quality-scan): the comment originally cited `search_symbols` snapshot files as ground truth for field order. The insta harness alphabetizes JSON keys via `parsed_sorted` before recording, so the snapshots are not declaration-order references. Doc-comment updated to make the struct itself the authoritative source.
